# 场景混剪使用文档

这份文档专门说明本项目新增的“场景混剪”能力，也就是通过 `scenes[]` 参数，把多个镜头自动解析成一条完整的 IMS 时间线。

它适合下面这类业务界面：

- 一个视频由多个镜头组成
- 每个镜头可以放一个或多个素材
- 每个镜头可以单独配置字幕、花字、配音
- 镜头之间和镜头内部素材之间可以配置转场

## 1. 能力入口

当前推荐的调用入口有两个：

- 模板入口：`Hunjian\AliyunImsMixcut\Template\SceneMixcutTemplate`
- 服务入口：`Hunjian\AliyunImsMixcut\Service\MediaProducingService::submitSceneMixcut()`

如果你只是想拿到构建后的时间线结构，可以直接使用 `SceneMixcutTemplate`。

如果你想直接提交到 IMS，优先使用 `submitSceneMixcut()`。

## 2. 设计思路

这套能力不是把前端参数直接硬塞给 IMS，而是先经过三层处理：

- `ScenePayloadNormalizer`
  负责参数归一化、补默认值、字段兼容和结构校验
- `SceneDurationResolver`
  负责计算每个镜头的最终时长
- `SceneTimelineAssembler`
  负责把镜头、素材、字幕、花字、配音装配成 `Timeline`

这样做的好处是：

- 前端可以传更贴近编辑器的数据结构
- 后端可以统一处理时间、素材、字幕绑定关系
- 后面字段变化时，优先改 normalizer 即可，不需要改整套模板

## 3. 基本调用示例

```php

use Hunjian\AliyunImsMixcut\Config\ImsConfig;

                    'materialId' => 'm-2',
```

## 4. 顶层参数说明

### 4.1 `canvas`

用于指定输出画布尺寸，不传时默认：

```php
array(
    'width' => 1080,
    'height' => 1920,
)
```

### 4.2 `outputMediaConfig`

建议显式传入 `OutputMediaConfig::oss(...)`，这样输出地址最清晰。

### 4.3 `global`

全局配置会作为默认值下沉到镜头或字幕层，常用字段有：

- `bgm`
- `watermark`
- `sceneTransition`
- `subtitleStyle`
- `wordArtStyle`

## 5. `scenes[]` 结构说明

每个镜头都是一个数组对象，常用字段如下：

- `sceneId`
- `sceneDuration`
- `transition`
- `materials`
- `dubbing`
- `subtitles`
- `wordArts`

### 5.1 `sceneDuration`

镜头最终时长优先级如下：

1. `sceneDuration`
2. `dubbing.duration`
3. 镜头内 `materials[].duration` 之和
4. 默认回退值 `3.0`

第一版不会主动探测远程音频或视频的真实媒体时长，所以如果你希望镜头时长跟配音走，请显式传 `dubbing.duration`。

### 5.2 `materials[]`

每个素材至少需要：

- `type`
  目前支持 `video`、`image`
- `url`

常用扩展字段：

- `materialId`
- `duration`
- `sourceRange`
- `sceneRange`
- `zOrder`
- `layer`
- `layout`
- `transition`

#### `sourceRange`

表示源素材裁剪区间：

```php
array(
    'in' => 0.0,
    'out' => 1.5,
)
```

#### `sceneRange`

表示素材在镜头内部的相对时间区间：

```php
array(
    'start' => 0.0,
    'end' => 2.0,
)
```

如果不传 `sceneRange`，后端会按素材数组顺序依次铺排。

如果你使用了手动 `sceneRange`，当前版本支持两种模式：

- 同一个镜头内所有素材都显式传 `sceneRange`
- 素材时间不能超出镜头总时长

### 5.3 `zOrder` / `layer`

当前版本已经支持镜头内素材层级。

你可以在素材上配置：

```php
array(
    'zOrder' => 10,
)
```

或者：

```php
array(
    'layer' => 10,
)
```

两者含义一致，都会映射到 IMS 里的 `VideoTrackClip.ZOrder`。

规则如下：

- 数字越大，层级越高
- 同一镜头内素材允许时间重叠
- 时间重叠时，画面前后关系由 `zOrder/layer` 控制
- 如果不传 `zOrder/layer`，后端不会强制写默认层级

这意味着你现在可以做：

- 背景全屏视频 + 前景人物视频
- 底图 + 浮层图片
- PIP 小窗画面
- 贴角标样式的视频/图片叠层

### 5.4 `dubbing`

配音支持两种模式：

1. 传已合成好的音频 URL
2. 传文案和音色，后端转为 `AI_TTS`

字段优先级如下：

- 如果有 `audioUrl`，优先使用普通音频 clip
- 如果没有 `audioUrl`，但有 `text + voice`，回退到 `AI_TTS`

常用字段：

- `audioUrl`
- `text`
- `voice`
- `duration`
- `speechRate`
- `pitchRate`
- `ssml`

### 5.5 `subtitles`

普通字幕使用独立字幕轨，常用字段：

- `text`
- `start`
- `end`
- `referenceMaterialId`
- `layout`
- `style`

这里的 `start / end` 都是“镜头内相对时间”，不是整条视频的绝对时间。

### 5.6 `wordArts`

花字和普通字幕分开建模，也会进入独立字幕轨。

第一版的实现方式是：

- 底层仍然使用 `SubtitleTrackClip`
- 但样式使用独立的 `wordArtStyle`
- 支持单条花字覆盖位置、颜色、描边和背景框

## 6. 转场规则

当前支持三层转场来源，优先级从高到低如下：

1. `materials[].transition`
2. `scenes[].transition`
3. `global.sceneTransition`

实现规则：

- 镜头内部素材优先用素材自己的转场
- 镜头入口素材可以继承镜头级转场
- 如果镜头和素材都没配，才回退到全局默认转场

当前转场按片段交叉淡化理解，不会自动延长镜头总时长。

## 7. 时间和引用关系

这套能力里最重要的两个概念是：

- 镜头内相对时间
- `referenceMaterialId`
- 素材层级

### 7.1 相对时间

前端传入的字幕、花字、素材 `sceneRange`，都应该使用镜头内时间。

例如：

- 第一镜头总时长 3 秒
- 花字 `start=1.0 end=2.2`

那它表示的是“在第一镜头内部从 1.0 秒到 2.2 秒出现”，后端会自动换算成整条视频的绝对时间。

### 7.2 `referenceMaterialId`

字幕不是直接引用 IMS 原始 clip ID，而是先引用前端传入的 `materialId`：

```php
array(
    'referenceMaterialId' => 'm-1',
)
```

后端会自动把它映射成最终的 `referenceClipId`。

### 7.3 素材层级

如果一个镜头里有多个素材时间重叠：

- `sceneRange` 决定“什么时候出现”
- `layout` 决定“出现在画面哪里”
- `zOrder/layer` 决定“谁压在谁上面”

例如：

- 背景素材：`sceneRange=0~4`，`zOrder=1`
- 前景素材：`sceneRange=1~3`，`zOrder=10`

那最终效果就是：

- 0~1 秒只显示背景
- 1~3 秒背景和前景同时显示，但前景在上层
- 3~4 秒恢复只显示背景

## 8. 校验规则

当前版本会对下面这些问题直接抛错：

- 没有 `scenes`
- 镜头里没有 `materials`
- `material.type` 不是 `video` 或 `image`
- `material.url` 为空
- `dubbing` 既没有 `audioUrl`，也没有 `text + voice`
- `start >= end`
- `referenceMaterialId` 在当前镜头里找不到
- 字幕、花字、素材超出镜头总时长
- 转场时长大于素材时长

异常类型为：

`Hunjian\AliyunImsMixcut\Exception\InvalidSceneMixcutException`

它会带上机器可读字段，例如：

```php
array(
    'code' => 'INVALID_SCENE_ITEM_RANGE',
    'message' => 'Scene item timing must stay inside resolved scene duration.',
    'path' => 'scenes[0].subtitles[0].end',
)
```

这样前端更容易定位到具体是第几个镜头、哪一条字幕出了问题。

## 9. 当前版本边界

为了先把主链路跑通，第一版刻意没有做下面这些能力：

- 自动探测远程媒体真实时长
- 自动切字幕句子
- 镜头内复杂并行多视频轨布局
- 智能补时长、补空白、补冻结帧
- 独立 sticker 元素模型

如果后面要扩展，建议优先往这些方向演进：

- `materials[].effects[]`
- `scene.effects[]`
- 真正的贴纸元素
- 自动混音和压 BGM
- 媒体元数据探测器

## 10. 推荐接入方式

如果你现在是做前端编辑器或后端接口，推荐这样接：

1. 前端统一维护 `scenes[]` 数据结构
2. 所有字幕、花字、素材时间都用镜头内相对时间
3. 所有素材都尽量带 `materialId`
4. 如果镜头时长要跟配音走，显式传 `sceneDuration` 或 `dubbing.duration`
5. 后端直接调用 `submitSceneMixcut()`

如果只是本地调试，也可以先用 `SceneMixcutTemplate::build()` 看输出的 `timeline->toArray()`，确认字段是否符合预期，再接真实提交。
