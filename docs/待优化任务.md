# aliyun-ims-mixcut 待优化任务

这份文档用于记录当前版本暂不立即实现、但已经明确有业务价值的后续优化项。

记录时间：2026-04-11

## 当前优先项

### 1. 增加回调 / 事件层

状态：
暂缓实现，进入待优化清单。

背景：
当前包已经具备：

- `ImsApplication`
- `MediaProducingService`
- `BatchProducingService`
- `CampaignProducingService`
- `JobRecord` / `CampaignRunReport`
- 文件仓储、导出器、应用装配层

但“任务提交前后、任务完成、任务失败、结果持久化完成”这些节点，仍然需要业务项目自己在外部串联日志、通知、审计、埋点和告警逻辑。

优化目标：

- 提供一层轻量、可插拔、不绑定框架的事件机制
- 让业务项目可以在关键生命周期节点挂接自定义逻辑
- 保持 PHP 7.3 兼容
- 不破坏当前 Builder / Service / Repository 结构

建议范围：

- 增加事件对象或上下文对象，至少覆盖：
  - 提交前
  - 提交后
  - 单任务完成
  - 单任务失败
  - 活动运行完成
  - 报告持久化完成
- 支持在 `ImsApplication` 层统一注册 listener
- 支持在 `CampaignProducingService` 或 `MediaProducingService` 中触发事件
- listener 能拿到：
  - `CampaignPlan`
  - `BatchTask`
  - `JobResult`
  - `ProduceResult`
  - `JobRecord`
  - `CampaignRunReport`
  - 业务 metadata

建议不要首版就做的内容：

- 不直接绑定 Laravel / Symfony 事件总线
- 不一开始就引入异步消息队列
- 不先做过重的事件订阅配置系统
- 不把所有逻辑都下沉成复杂中间件链

建议实现方式：

- 先做同步事件分发器
- 提供简单接口，例如：
  - `EventDispatcherInterface`
  - `NullEventDispatcher`
  - `InMemoryEventDispatcher`
  - `EventListenerInterface`
- 在 `ImsApplicationFactory` 中允许注入 dispatcher
- 默认无副作用，不配置 listener 时不影响现有行为

预期收益：

- 业务项目更容易接日志、告警、消息通知
- 更容易做任务审计和全链路追踪
- 更容易在不改核心服务逻辑的情况下扩展行为
- 更适合作为真实生产项目中的接入 SDK

验收标准：

- 不修改现有公开 API 的既有语义
- 不引入 PHP 8+ 语法
- smoke test 继续通过
- 至少覆盖一条“活动运行完成后自动触发持久化/通知”的示例
- README 或使用文档新增事件层接入说明
